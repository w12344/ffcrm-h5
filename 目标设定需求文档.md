## 🎯 需求概述

### 1.1 背景与问题

**业务背景**
- CRM系统不仅需要记录过程和结果数据，还需要提供一个“指南针”。在每年度初，管理层需要为团队制定核心的年度战略目标（如：年度总营收金额、年度总入学人数、年度总线索获取量）。
- 制定年度目标后，需将大盘指标科学、公平地拆解到各个团队成员（作战单元）身上，以便于后续在业绩看板中进行实际达成率（进度条）的追踪。

**用户痛点**
- **痛点1：目标分配与总盘脱节**：过去管理者使用Excel分发目标，很难直观看到“当前分配给各员工的业绩总和，是否已经达到了团队总KPI？”（覆盖率黑盒）。
- **痛点2：跨角色指标不统一**：销售关注“钱”和“成单”，市场关注“线索量”，一套固定维度的目标系统无法满足多部门需求。
- **痛点3：沟通与下发成本高**：目标分配完后，缺少系统级的“一键下发”和备注说明（如特殊政策倾斜），导致一线员工对目标的理解有偏差。

**机会点**
- 提供一个支持“实时计算分配覆盖率”、“双视角（销售/市场）切换”的动态分配工作台，将目标从“数字”变成可跟踪的“系统基线”。

### 1.2 需求范围

**本期包含**
- ✅ 多角色权限视图感知（Mock支持 Sales / Marketing 视图切换）
- ✅ 团队三大核心战略目标设置（总营收目标、总入学目标、总线索目标）
- ✅ 目标拆解与实时指标反馈（分配总额、分配覆盖率、人均目标值）
- ✅ 团队成员目标分配列表（按人头精确分配 `targetAmount`、`targetEnrollment`、`targetLeads`）
- ✅ 团队成员目标备注功能（针对单人的补充说明）
- ✅ 战略目标一键发布（模拟消息通知与落库动作）

## 🔄 业务数据闭环链路

战略目标设定的业务逻辑是整个CRM数据考核的**起始点**：
1. **大盘设定（定调）**：管理层在 `GoalOverview` 中输入当前周期的团队总目标（如 100万营收，50个入学）。
2. **目标拆解（派兵）**：在 `GoalList` 列表中，根据各成员的历史能力与负责渠道，分别填入具体的子目标。
3. **动态校验（对齐）**：系统利用 `useMemo` 实时聚合所有成员的子目标之和，计算**分配覆盖率**。若覆盖率 < 100%，意味着大盘目标未分完；若 > 100%，则代表团队超额领任。
4. **发布下达（誓师）**：点击保存发布，系统展示“战略目标已发布至各作战单元！”，将目标数据写入后端。
5. **执行监控（闭环）**：后续在 Dashboard 首页，各个员工/团队的实际完成数据将与此处的设定的目标进行对比，计算完成率。

## 🎨 核心页面与功能设计

### 4.1 核心页面说明 (`GoalSetting/index.tsx`)

**核心组件 1：全局目标看板 (`GoalOverview`)**
- **核心元素**：团队总目标输入框（支持三大核心指标）、实时动态分配仪表盘。
- **交互逻辑**：
  - 允许负责人直接编辑修改大盘的 `totalGoal`（总营收）、`totalEnrollmentGoal`（总入学）、`totalLeadsGoal`（总线索）。
  - 根据下方列表成员的分配数值，实时渲染：已分配总额（Assigned）、分配覆盖率（Coverage Rate）、人均目标（Avg Goal）。
  - **角色联动**：如果是 `userRole === 'marketing'`，则大盘重点展示“总线索目标”；如果是 `sales`，则重点展示“营收与入学”。

**核心组件 2：成员目标分配列表 (`GoalList`)**
- **核心元素**：团队成员卡片/列表，包含成员基础信息（头像、姓名、岗位）、三大目标输入框、备注输入框。
- **交互逻辑**：
  - **数值录入**：触发 `handleUpdateTarget`，支持对 `targetAmount`, `targetEnrollment`, `targetLeads` 进行精确到人的数值更新。
  - **备注说明**：触发 `handleUpdateRemark`，支持针对个别员工（如新手期员工）填写保护政策或特殊说明（如：“重点突破抖音渠道”）。
  - **正则拦截**：数字输入框支持正则替换 `val.replace(/[^\d]/g, '')`，防止非数字字符导致系统计算 NaN 崩溃。
  - **提交发布**：底部“保存并发布”按钮，触发 `handleSave` 动作。

## 🔧 详细设计与数据字段字典

为了确保前端状态与大盘计算逻辑一致，以下为本页面核心的数据状态（State）及派生指标（Derived Metrics）计算规则。

### 7.1 核心基础数据 (Base State)

| 字段名称 | 字段变量名 | 数据类型 | 业务意义 | 初始值来源 |
| --- | --- | --- | --- | --- |
| **总营收目标** | `totalGoal` | Number | 团队周期内的总销售额（GMV）KPI | `TEAM_STAT_CONFIG.totalGoal` |
| **总入学目标** | `totalEnrollmentGoal` | Number | 团队周期内的总入学人数（转化数）KPI | `TEAM_STAT_CONFIG.totalEnrollmentGoal` |
| **总线索目标** | `totalLeadsGoal` | Number | 团队周期内的需获取的线索总数 KPI | `TEAM_STAT_CONFIG.totalLeadsGoal` |
| **成员列表** | `members` | Array<ItemType> | 当前团队内的所有可用员工列表及其目标 | `MOCK_TEAM_MEMBERS` |
| **当前角色** | `userRole` | Enum | 当前操作者的权限视角（'sales'/'marketing'）| 默认为 'sales' (后续对接登录用户信息) |

### 7.2 成员分配字段结构 (`ItemType`)

| 字段名 (Key) | 界面名称 | 数据类型 | 业务流转与说明 |
| --- | --- | --- | --- |
| `id` | 员工ID | String | 唯一标识，更新状态时的 Key |
| `name` | 员工姓名 | String | 展示用 |
| `targetAmount` | 分配营收目标 | Number | 销售角色核心考核点，参与覆盖率计算 |
| `targetEnrollment`| 分配入学目标 | Number | 销售角色核心考核点，参与覆盖率计算 |
| `targetLeads` | 分配线索目标 | Number | 市场角色核心考核点，参与覆盖率计算 |
| `remark` | 备注说明 | String | 负责人下达的专项策略与保护说明 |

### 7.3 核心指标动态计算公式 (`GoalMetrics`)

系统使用 `useMemo` 对以下指标进行实时联动计算：

| 派生指标名称 | 变量名 | 计算逻辑与公式 | 业务意义 |
| --- | --- | --- | --- |
| **已分配营收总额** | `assignedAmount` | `Σ(members[i].targetAmount)` | 汇总所有员工背负的营收KPI |
| **已分配入学总数** | `assignedEnrollment`| `Σ(members[i].targetEnrollment)` | 汇总所有员工背负的入学KPI |
| **已分配线索总数** | `assignedLeads` | `Σ(members[i].targetLeads)` | 汇总所有员工背负的线索KPI |
| **分配覆盖率(营收)**| `coverageRate` | `(assignedAmount / totalGoal) * 100` 四舍五入 | 衡量团队目标是否100%被员工认领 |
| **分配覆盖率(入学)**| `coverageRateEnrollment`| `(assignedEnrollment / totalEnrollmentGoal) * 100` | 同上 |
| **分配覆盖率(线索)**| `coverageRateLeads` | `(assignedLeads / totalLeadsGoal) * 100` | 同上 |
| **人均目标** | `avgGoal` / `avgEnrollmentGoal` / `avgLeadsGoal` | `已分配总数 / members.length` | 辅助管理者判断当前的分配是否均匀合理 |

## ⚠️ 异常处理与兜底机制

### 10.1 异常场景清单

| 场景编号 | 异常场景 | 触发条件 | 用户感知 | 系统处理 |
| --- | --- | --- | --- | --- |
| E-001 | 非法字符输入 | 在目标输入框输入字母或特殊符号 | 输入框不响应非法字符 | `handleUpdateTarget` 中使用正则 `/[^\d]/g` 进行拦截过滤，若为空则回退至 `0`。 |
| E-002 | 大盘目标被设为0 | 用户将团队总目标清空或置为 0 | 覆盖率不报错 | 计算覆盖率时采用兜底逻辑：`(assignedAmount / (totalGoal \|\| 1))`，避免除以零导致 Infinity 或系统崩溃。 |
| E-003 | 覆盖率超出100% | 员工认领的KPI总额超过了大盘总KPI | 覆盖率进度条可能变色或超限 | 允许覆盖率 > 100%（超额分配是常见业务场景），但前端UI需做超额样式的兼容（如进度条拉满并变为绿色/红色，视组件而定）。 |
| E-004 | 团队成员为空 | 初始化获取不到成员列表 | 提示“暂无成员”或人均目标计算不报错 | `avgGoal` 等人均计算时采用兜底逻辑：`(assignedCount \|\| 1)`，防止空数组导致除零错误。 |
